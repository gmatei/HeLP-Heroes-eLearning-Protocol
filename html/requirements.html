1. Introduction

1.1 Purpose

This SRS describes “Heroes: eLearning Protocol” (HeLP), the web-application software in its first version 1.0, elaborating on the entire scope of the software.

1.2 Document Conventions

Each requierment has an ID according to the following schema: [USR | SW] - REQ [NUMBER], where USR strands for user requierments, SW stands for software requierments and NUMBER is the invididual number associated with it.

1.3 Product scope

The software product is intended for teenagers and young adults, but could be expanded to any type of person wanting to learn new information in a friendly manner, aided by its favorite superhero. Users are able to acquire knowledge in different domains, each of them associated to a superhero. After mastering the knowledge in the specific domain, they can head into a trivia quiz mode where they can use the information learned and the cards won through learning to achieve a score as high as possible so they can be featured on the leaderboard.

2. Overall description

2.1 Product Perspective

"Heroes: eLearning Protocol" (HeLP) is a a new, stand-alone product. It's main selling point is having comic book characters assist the process of learning. As such, it will provide the user with a fun and interactive way to learn about school subjects, and apply the knowledge in a survival type of game based on which users will be ranked. 2.2 Product Functions

The following sub-sections describe the basic functionality of the final product.

2.2.1 Main Module

The main module will serve as the main controller for all of the other modules. It will be the place the user arrives at upon login/registration, and will provide links to the practice, wiki and arena modules.

2.2.2 Login Module

The login module will serve as a means for the user to log into an already existing account, or create one otherwise.

2.2.3 Learning Module

The learning module will serve as a means for the user to learn about certain subjects or practice already mastered ones.

2.2.4 Wiki Module

The wiki will provide information about the subject associated with the hero chosen by the user.

2.2.5 Arena Module

The arena module will entertain a survival mode consisting of at least 100 questions for the user, which will choose a character to help them.

2.3 User Classes and Characteristics

The application is built with the following user classes in mind:

  1. End users by interests:
    • Children: learning about superheroes; will make use only of the wiki module
    • Teenagers (main target): learning about schools subjects; will make use of the entire application
    • Parents: supervising the usage of the application by their children; will make use of the entire application to familiarize themselves with the content viewed by their children
    • Admins: management of the data used and displayed by the application; will only use managerial functions
  2. End users by type of device used:
    • Desktop users (main target): content will be more layed-out and easier to use
    • Laptop/Tablet users (main target): content will be slightly less layed-out than for desktop users
    • Mobile users: content will be more compact; some features will require landscape orientation

2.4 Operating Environment

The environment in which the software will operate is any environment which supports a web browser (i.e. Chrome, Firefox, Safari, Opera etc.).

2.5 Design and Implementation Constraints

Following is a list specifying the overhead taken into consideration as of the writing of the requirements:

  • Intermediary knowledge of JavaScript/PHP for the back-end
  • Basic/Intermediary knowledge of HTML5/CSS3 for the front-end
  • A working connection to a SQL database
  • Communication through a RESTful API (using the HTTP protocol)

3.3 Software requirements

Listed below are the software requirements, indicating the underlying behaviour on the server side.

3.3.1 Querying the database

User authentication
User credentials will be POSTed to the server, which will query the database regarding the received information. In turn, the database shall respond with authentication success or failure.
Learning area - character selection
The server will query the database for all the available characters. In turn, the response body shall consist of a list of all the characters in JSON format.
Battle area - character selection
The server will query the database for all the unlocked characters. In turn, the response body shall consist of a list of all the unlocked characters by the player in JSON format.
Learning mode - quiz
The server will query the database for 10 questions at random, chosen from the character's associated domain. In turn, the response body shall consist of a list of questions in JSON format.
Battle arena (on quiz begin)
The server will query the database for 100 questions at random. In turn, the response body shall consist of a list of questions in JSON format.
Battle arena (on quiz end)
Upon answering all the questions or running out of hearts, the server will insert into the database a new entry consisting of the name of the player, the score and the character they played with.
Leaderboard
Upon entering the leaderboard page, the server will query the database for the top 100 best scores, along with the player usernames and the character with which they played.
Wiki (portal)
Upon entering the wiki portal page, the server will query the database for a list with all the characters. In turn, the response body shall consist of a list of all the characters in JSON format.
Wiki (individual page)
Upon entering an individual character's wiki page, the server will query the database for the character's information, which will be granted by the database in JSON format.

3.3.2 Handling the data received from the database

Authentication response
On success, the user is logged in and redirected to the main page.
On failure, the user will be prompted to fill in their credentials again.
Main page - character list
Upon receiving the character list in JSON format from the database, the server will store it in a const array and will use it for the learning area (where all the heroes can be cyckled through) and battle area (where only unlocked heroes can be cycled through).
Learning mode - quiz start
Upon receiving a list containing 10 questions in JSON format from the database, the server will store it as a const array and will construct and display the questions together with their answers.
Upon constructing the quiz, the correct answers will be marked, and they should only be revealed when the user submits their answers.
If the answers were not 100% correct, the user will be given the opportunity to retry the quiz with a different set of questions.
Battle mode
Upon receiving a list containing 100 questions in JSON format from the database, the server will store it as a const array and will display one question at a time.
When the user answers the current question and clicks on the `Next question` button, the next question will be loaded onto the screen. If the last question was reached, a victory screen should be displayed instead.
If the last remaining life has been depleted, a game over screen shall be displayed onto the screen.
Wiki portal
Upon receiving a list containing all the characters in JSON format from the database, the server will store it as a const array and will display 6-8 heroes on the page.
The current selection of characters will be changed when the user clicks on the `Next` or `Previous` buttons.
Wiki individual entry
Upon receiving the data corresponding to the chosen character in JSON format from the database, the server will construct the character's individual wiki page, consisiting of the character's image (left side) and information regarding the subject associated with the character (right side; text will be wrapped around the image).
The next/previous character individual pages will be constructed in similar fashion when the user clicks on the respective buttons at the bottom of the page.

3.4 Communication requirements

As stated in earlier requirements, the end user will need acces to a web browser in order for the application to be displayed. Communication between the client and the server will be made through the use of the HTTP communication standard. Data which will be manipulated by the back-end will be JSON compliant (e.g. key-value pairs). Dynamically changing content will be handled according to Ajax norms.
