In the beginning, there was nothing... Then an idea came.

1. First steps

The project started as a necessary requierment for the final grade of the Web Technologies course around the fourth week and has since developt into something unique. At first, the team (meaning Goanță Matei, Vlad Theodor and Petercă Adrian) were given a list of themes to work around, each one of them being described briefly, but with no exact specifications or limitations. Because of the interest it presented, the SuH (SuperHero Web Game) theme was chosen and so began a journey with many twists and turns, but with a final destination filled with knowledge and accomplishment.

1.1. Tehnical details

Regarding the tehnical details, the team used an Apache environment, because of its ease of installation and usability, alongside the bountiful information available online.

For the storage part (this includes user credentials, heroes information, leaderboard statistics etc.) a MariaDB was used at first (because it came with the XAMPP installation) but was quickly changed to a PostgreSQL cloud-deployed-database instead. The motive was quite simple: because of the frequent changes requiered to the database (insertions, deletions) for testing purposes, it was better to be available deployed on the cloud rather than stored locally and updated every time someone makes a change.

The team shared their code using GitHub, but for the moment the repository is marked as private. The commits were made using a standard protocol (meaning that every commit had to be in English and the summary had to be precise) for ease of use and development in the future.

2. The first complete view of the webpage

Up until week 8, the project had to have at least the front-end part complete, with all the subpages, stylesheets and other details available on GitHub. The team worked tirelessly and had the following scheme to guide themselves:

Decide what pages the app will have: the login interface, the register interface, the main page, pages for every hero etc.
This step did took some time and careful planning, because little mistakes and errors here meant that the project may fail in the future. With that in mind, the team settled for the following set of pages: the login/register interface, the main page, the leaderboard page, the battle mode / practice mode interface, the wiki page alongside a page dedicated to every hero, the requierments page and a more administrative-orientated page, the admin interface.
Write the HTML and CSS following a simple theme, with simple colors and fonts.
Each page was requiered to follow the coding standards and, besides those, the team also set up a temporarily UI/UX theme to work with.
After most of the pages were done, integrate them only on the front-end level.
A bit of JavaScript was requiered here to make simple, hardcoded buttons behave nicely and give the user a feeling of an overall fully developt website.
After the pages were intergrated and tested, apply the same UI/UX theme for the whole website.
The final step was synchronizing all pages to have the same theme and to appeal to a standard user. The buttons had animation added, the images were made to fit different screens, the text received a new font, etc.

3. The back-end part

After the eighth week, the team had up until the evaluation period (which was about eight weeks) to build the back-end part of the system, integrate it, test it and make it look really nice. And it did!

The first step was to design and implement the database. Up until this point, there was a general idea of what tables it should have, but no strict definition were given to them (because there was no need to). A user should have a username, a password and some other information, but what else? So, finally, five tables were created: users, admins, heroes, leaderboard and users_heroes_list. Later on, two more will be added: sessions and sessions_admin for authentication purposes and session validation.

After the database was created and deployed on cloud, a REST API was created to send HTTP requests from the client. The API is written in PHP, while the client is mostly written in JavaScript.

For the user validation part, cookies were used. This meant that, when a user successfully logs in, a token of 128 characters gets created on the server, which will have to be sent (alongside the user's id) with each request from the client. The client will receive this token and store it in a cookie, up until the user logs out.

As for the website functionality, the leaderboard was made available in a RSS feed, the battle mode and practice mode were implemented and integrated fully using a dynamic content generation approach, meaning that for each user different content (which is the necessary approach for a contest) will be generated.

For the administrative part, a special admin account was created. With it, the following functionalities are available:

  • setting the ban status of a user
  • adding a new question into the database
  • adding a new hero into the database
  • registering a new admin user
